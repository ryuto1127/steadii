# Steadii — Product Requirements Document

## 1. プロダクト概要

### 1.1 一文定義

Steadiiは、大学生のアカデミックライフ全体をAIエージェントで管理・支援するWebアプリケーション。Notion・Google Calendarと連携し、チャットを中心とした会話インターフェースを通じて、情報の記録・参照・解説・予定管理を横串で実行する。

### 1.2 コアバリュー

大学生のアカデミックな管理負荷を下げ、勉強そのものに集中できる状態を作る。

### 1.3 なぜ今これを作るか

現状、大学生は以下のツールを手動で行き来している:

- シラバスPDF(ダウンロードフォルダに放置)
- Notion(授業ノート、課題管理)
- Google Calendar(授業、締切、予定)
- スマホのカメラロール(間違えた問題のスクショ)
- ChatGPT等(わからない問題の質問)

これらが分断されているため、「この授業を休むとどうなる?」「この問題と似た問題を前に解いた気がする」といった自然な問いに対して、ユーザーは自分で複数ツールを横断して答えを探すことになる。Steadiiは、この分断を埋めるAgent中心のインターフェースを提供する。

### 1.4 対象ユーザー

- 大学生(北米・日本を主眼、日英バイリンガル対応)
- Notionをすでに使っている、または使う意思のあるユーザー
- α版は招待制10人前後、将来的に一般公開

### 1.5 スコープ外

- 高校生以下向け機能
- 教員・TA向け機能
- クラスメイト間のコラボレーション機能(Post-v1.0)
- 間違いノートの間隔反復(SRS)機能(Post-v1.0)
- モバイルネイティブアプリ(PWA化はPost-v1.0)

## 2. コア体験

### 2.1 Chat中心の設計思想

Steadiiの顔はチャット画面。ユーザーは原則としてエージェントに話しかけることで、あらゆる機能を呼び出せる。各機能には専用タブも存在するが、それらは「結果の閲覧・整理」用であり、入力の主経路はChat。

理由: 機能数が増えるほど、個別UIを記憶する負荷が上がる。チャット1箇所で済ませられることで、ユーザーの認知負荷を最小化する。

### 2.2 Agentの振る舞い

- **言語**: ユーザーが使った言語で応答(日本語で質問 → 日本語で回答、英語で質問 → 英語で回答)
- **Streaming**: 常にstreamingで即時応答を開始
- **ツール呼び出し中の可視化**: 「Notionにページ作成中...」のような中間状態を表示
- **永続的な権限**: 一度セットアップで許可したリソース(Notionワークスペース、Calendar、登録済みリソース)は、Agentが自由に読み書きできる
- **破壊的操作のみ確認**: ページ削除、共有設定変更、大量データ編集は実行前に確認。作成・更新・閲覧は自動実行

## 3. 機能要件

### 3.1 認証・アカウント管理

#### 3.1.1 Google OAuth ログイン

- 唯一の認証方式はGoogle OAuth
- 初回ログイン時、自動的にアカウントを作成
- Googleアカウントの言語設定から初期UI言語を推定

#### 3.1.2 アカウント情報

- プロフィール: 名前、メールアドレス、プロフィール画像(Googleから取得)
- 設定: UI言語、Agent確認挙動、通知設定

### 3.2 初回セットアップ

新規ユーザーは以下のフローを完了する必要がある:

1. Google OAuthでサインイン(言語は自動判定、後から変更可)
2. **Notionワークスペースを接続**(必須)
3. **Google Calendarを接続**(必須)
4. 自動セットアップ実行:
   - Notionに「Steadii」という親ページを作成
   - 親ページの下に「間違いノート」データベースを作成(クラス別リレーション対応)
   - 親ページの下に「課題」データベースを作成
   - 親ページの下に「シラバス」データベースを作成
5. 任意: 既存のNotionページ・DBを「登録済みリソース」として追加
6. 完了

### 3.3 統合チャット(Agent)

#### 3.3.1 基本仕様

- マルチターン会話、履歴は自前DBに保存
- 履歴は後から検索・参照可能
- 会話ごとにタイトルを自動生成(Nanoモデル使用)
- 添付ファイル対応: 画像(スクショ)、PDF

#### 3.3.2 Agentツール一覧

Agentは以下のツールを呼び出せる:

**Notion関連**
- Notionページを検索
- Notionページを取得(コンテンツ読み込み)
- Notionページを作成(親ページ指定可能)
- Notionページを更新(追記・書き換え)
- Notionページを削除(**確認必須**)
- NotionデータベースのレコードをCRUD

**Google Calendar関連**
- イベント一覧取得(期間指定可)
- イベント作成
- イベント更新
- イベント削除(**確認必須**)

**内部機能**
- シラバスPDF/画像を解析
- 間違い問題をstep-by-step解説
- ユーザーの登録リソース一覧を取得
- チャット履歴を検索

#### 3.3.3 言語対応

- UIは日本語・英語を切り替え可能
- Agentの応答はユーザーの入力言語に合わせて自動切替
- 1つの会話の中で言語を混ぜても自然に動作

### 3.4 Notion連携

#### 3.4.1 接続

- Notion OAuth 2.0で接続
- ユーザーが許可したワークスペースに対して読み書き可能
- アクセストークンは暗号化して保存

#### 3.4.2 リソース登録モデル

Agentは「登録されたリソース」に対してのみ能動的にアクセスする:

- 初回セットアップで自動作成される「Steadii」親ページとその配下は自動登録
- ユーザーが任意のNotionページ・DBを追加登録可能
- 登録リソースは設定画面で一覧・削除可能
- リソースタイプに制限なし(授業・課題・研究ノート・読書記録など、何でも可)

### 3.5 Classes(授業)

Steadiiのデータモデルは**授業(Class)を中心**に回る。間違いノート・課題・シラバスはすべて授業に紐づき、授業ページを開けばその授業に関するすべての記録が一覧できる。

#### 3.5.1 Classesデータベース(Notion)

Notionの「Steadii/Classes」データベースに保存。プロパティ:

- **Name**(title): 授業名(例: 「Intro to Computer Science」)
- **Code**(rich_text): 授業コード(例: 「CSC101H1」)
- **Term**(select): 学期(例: 「Fall 2026」「Winter 2027」)
- **Professor**(rich_text): 担当教員名
- **Color**(select): カラー(blue / green / orange / purple / red / gray / brown / pink)
- **Status**(select): ライフサイクル状態(active / archived、デフォルトは active)

#### 3.5.2 ライフサイクル

- 新規追加: active状態で作成される
- アーカイブ: 学期終了などのタイミングでユーザーがarchivedに手動移行、またはAgentが「この授業はもう終わった」と会話で指示されたら提案
- Classesレコード自体は削除せず、archivedは一覧のデフォルトフィルタから外れる

#### 3.5.3 他DBとのリレーション

間違いノート・課題・シラバスの各データベースは、Classesへのrelationプロパティ「Class」を持つ(dual_propertyによる両方向同期 — Classes側の各レコードからも紐づく間違い・課題・シラバスが一覧できる)。

#### 3.5.4 Web UI の位置付け

Phase 2〜5では、授業の追加・編集・アーカイブはNotion上で直接行う(auto-discoveryで登録リソースに反映される)。専用の `/app/classes` UI(Web上での授業の作成・編集・アーカイブ)は **Phase 6 に延期**。

### 3.6 間違いノート

#### 3.6.1 入力方式

- Chatに問題のスクショ/画像を添付
- または専用タブから問題を追加(画像アップロード + テキスト記入)

#### 3.6.2 処理フロー

1. ユーザーが画像を添付して「この問題を解説して」等と投げる
2. Agentがstep-by-step解説を生成:
   - 問題の読み取り
   - 解き方のステップごとの説明
   - **「なぜこの視点が必要だったか」**のメタ解説
   - 「次に同じタイプの問題と出会ったら何を思い出すべきか」
3. Chat内に解説が表示される
4. ユーザーが「間違い集に追加」ボタンを押す
5. 保存ダイアログ: 授業、単元、難易度、任意タグを入力/選択
6. Notionに「Steadii/Mistake Notes」DB配下として保存

#### 3.6.3 Notion上のデータ構造

「Mistake Notes」データベースのプロパティ:
- Title(title): 問題の要約
- **Class**(relation → Classes): 関連する授業(dual_property)
- Unit(rich_text): 単元
- Difficulty(select): easy / medium / hard
- Tags(multi_select): 任意のタグ
- Date(date): 追加日
- Image(files): 元画像

ページ本文:
- 問題画像
- Agentが生成したstep-by-step解説
- メタ解説(なぜこの視点が必要だったか)
- 次回の学習ポイント

### 3.7 課題管理(簡易)

#### 3.7.1 データ構造

「Assignments」データベースのプロパティ:
- Title(title)
- **Class**(relation → Classes): 関連する授業(dual_property)
- Due(date): 締切
- Status(select): Not started / In progress / Done
- Priority(select): Low / Medium / High
- Notes(rich_text): 補足

#### 3.7.2 機能

- Web UI上で課題一覧表示(締切順、授業別)
- Chat経由で「次の締切は?」「物理の課題を追加」等の操作
- 課題の締切が近づいたらWeb UI上で強調表示(通知はPost-MVP)

### 3.8 シラバス取り込み

#### 3.8.1 入力方式

- PDF(アップロード)
- Webページ(URL入力、公開ページのみ)
- 画像/スクショ(アップロード、ペースト)

#### 3.8.2 処理フロー

1. ユーザーがChatに添付 or URLを投げる、あるいは専用UIでアップロード
2. 内部で自動判定:
   - 画像 → Vision APIで処理
   - PDF小(5ページ以下) → Vision APIで処理
   - PDF大 → テキスト抽出 + 表構造のあるページのみVision併用
   - Webページ → fetchしてHTML解析
3. 構造化抽出(科目名、担当教員、評価配分、欠席ルール、課題スケジュール、教科書、オフィスアワー等)
4. **プレビュー表示**(ユーザーが内容を確認)
5. 「保存」ボタンでNotionに保存(「Steadii/Syllabi」データベース配下)
6. 保存後、「このシラバスについて質問する」ボタンで新規チャット起動

#### 3.8.3 Notion上のデータ構造

「Syllabi」データベースのプロパティ:
- Title(title)
- **Class**(relation → Classes): 関連する授業(dual_property)
- Term(rich_text)
- Grading(rich_text)
- Attendance(rich_text)
- Textbooks(rich_text)
- OfficeHours(rich_text)
- SourceURL(url)

#### 3.8.4 保存後の活用

- 後日「このシラバスの欠席ルールを教えて」等の質問にAgentが回答
- 登録リソースに自動追加されるため、Agentは以降自由に参照可能

### 3.9 Auto-discovery(自動登録)

Steadii親ページ配下に追加されたNotionリソースは、ユーザーが明示的に登録しなくても**自動で認識・追跡**される。

#### 3.9.1 スコープ

- 監視対象: 「Steadii」親ページの直下の子ページ・子データベースのみ
- Steadii親ページの外にあるページは**従来どおりユーザーが手動で登録**(任意のNotionページ・DBを`/app/resources`から追加可能)

#### 3.9.2 仕組み

- `discoverResources(userId)` が Notion の `blocks.children.list` で親ページ配下を列挙し、`registered_resources` テーブルに差分を反映
- 新規検出: 未登録のページ / DBを `autoRegistered=true` で追加
- ステール検出: 以前存在していたが現在は親配下にない(Notion上で削除された or 移動した)リソースは `archived_at` を立ててソフトアーカイブ。ハード削除はしない(過去の会話履歴からの参照切れ防止)
- 60秒キャッシュ(ユーザー単位、メモリ内)でNotion APIへの連続リクエストを抑制

#### 3.9.3 トリガー

- Notion OAuth コールバック完了時(初回接続 & 再接続)
- Notion自動セットアップ完了時
- Agentのリソース一覧参照ツール呼び出しの直前(Phase 3以降、60秒キャッシュでスロットリング)
- `/app/resources` の「Refresh from Notion」ボタン押下時

### 3.10 Google Calendar連携

#### 3.10.1 接続

- Google OAuthのscopeで Calendar API を要求
- 複数カレンダー対応(ユーザーがデフォルトカレンダーを選択)

#### 3.10.2 Web UIでのカレンダー表示

- 今週ビュー、今月ビューを切り替え
- イベントをクリックで詳細表示
- 予定追加フォームあり(Agentでも追加可能)

#### 3.10.3 Chat経由の操作

- 「来週の火曜15時に数学の授業を追加」のような自然言語で予定操作
- 「今日の予定は?」で今日のイベント一覧取得

### 3.11 Web UI 構造

#### 3.11.1 ナビゲーション

サイドバー構成:
- **Chat**(トップ画面、会話一覧 + 現在の会話)
- **Calendar**(週・月ビュー)
- **Mistakes**(間違いノート一覧、Notionの「Mistake Notes」DBのミラー表示)
- **Syllabus**(シラバス一覧)
- **Assignments**(課題一覧)
- **Resources**(登録リソース管理)
- **Settings**(各種設定、使用量、課金管理)

なお Classes の専用UI(`/app/classes`)は Phase 6 に延期する(§3.5.4 参照)。

#### 3.11.2 表示の基本方針

Notionに保存されたデータは、Web UI側では**読み取り中心のミラー表示**を持つ:
- 一覧、検索、フィルタは可能
- 編集は「Notionで開く」ボタンで該当ページに遷移
- ただし間違いノート・課題・シラバスは軽量なインライン編集も可(Post-MVPで判断)

### 3.12 使用量と課金

#### 3.12.1 使用量表示

- 設定画面に「今月の使用量」バーを表示
- 単位は**Credits**(内部的なトークン換算、ユーザーには金額を見せない)
- プランの上限に対する割合で表示(例: 「今月の使用量 42%」)
- ドル表示・トークン数表示は一切しない

**Credits換算ルール**(内部仕様):
- 1 Credit ≈ $0.01 相当のトークン消費
- 無料プラン: 500 Credits/月($5相当)
- 有料プラン: 2000 Credits/月($20相当)
- 換算レートは内部定数として管理、将来変更可能

#### 3.12.2 プラン

| プラン | 月額 | 月間Credits |
|---|---|---|
| Free | $0 | 500 |
| Pro | $20 | 2000 |

#### 3.12.3 上限到達時の挙動

- 80%到達時: Web UIに警告バナー表示
- 100%到達時: チャット送信不可、「プランをアップグレード」または「来月まで待つ」の選択肢を表示
- 有料ユーザーの100%到達: 同様にブロック(今月分の追加購入はPost-MVP)

#### 3.12.4 決済

- Stripe Checkoutでサブスクリプション開始
- Stripe Customer Portalでプラン変更・解約
- Webhook経由でプラン変更を自前DBに反映

#### 3.12.5 Redeemコード

2種類のコードを用意:

**Adminコード**
- 適用すると月間Credits無制限
- 管理者本人のテスト用
- 複数生成可能、各コードは使い捨てor期間指定

**Friendコード**
- 適用すると有料プラン(2000 Credits)相当を指定期間(1ヶ月/3ヶ月/1年)付与
- 決済を経由しない
- 知人への配布用

Redeemコード入力UI:
- 設定画面に「コードを入力」フィールド
- 成功時に適用内容を表示
- 履歴を設定画面で確認可能

**Admin自身も通常アカウントでRedeemコードを使って「Admin」状態になる**(UX検証のため、特別なフラグ方式は採用しない)。

### 3.13 設定画面

- プロフィール情報(Googleから取得、編集は限定的)
- UI言語切替(日本語/英語)
- 接続済みサービス管理(Notion、Google Calendar の接続状態、再接続、切断)
- 登録済みリソース管理(Notionリソース一覧、追加、削除)
- Agent確認挙動設定:
  - 破壊的操作のみ確認(デフォルト)
  - すべての操作を確認
  - 確認なし(自動実行)
- 使用量表示(今月のCredits使用率)
- プラン管理(現在のプラン、アップグレード、Stripe Customer Portalへのリンク)
- Redeemコード入力・履歴

## 4. 非機能要件

### 4.1 パフォーマンス

- Chat初回トークン到達 < 1秒(streaming開始)
- ページ遷移 < 500ms
- Notion読み書きのレスポンス < 3秒(Notion API依存)

### 4.2 セキュリティ

- OAuth トークンは暗号化してDB保存(アプリケーションレベル暗号化)
- Stripe決済情報は一切アプリ側で保持しない
- HTTPS必須
- CSRF対策、Rate limiting

### 4.3 プライバシー

- ユーザーのNotionデータは会話の文脈として必要な範囲のみOpenAIに送信
- OpenAIへの送信データはデフォルトで学習には使われない(API仕様)
- ユーザーはアカウント削除時に全データを削除可能

### 4.4 可用性

- Vercelの無料枠で運用、α版期間は稼働率99%目標
- 外部APIの障害(Notion、Google、OpenAI)時はグレースフルに失敗、ユーザーに状況を説明

### 4.5 国際化(i18n)

- UI: 日本語、英語
- Agent応答: 日英自動切替
- 日付フォーマット、タイムゾーンはユーザーのロケール設定に従う

### 4.6 アクセシビリティ

- キーボードナビゲーション対応
- スクリーンリーダー対応(必須要素にaria-label)
- カラーコントラスト WCAG AA 準拠

## 5. モデル選定(OpenAI API)

ルールベースで以下のように振り分ける:

| タスク | モデル | 理由 |
|---|---|---|
| 通常チャット、ツール呼び出し | GPT-5.4 Mini | コスパ良好、ツール呼び出しに十分 |
| 間違いノートのstep-by-step解説 | GPT-5.4 | 品質が重要、視点の深さが必要 |
| シラバスのPDF構造化抽出 | GPT-5.4 | 複雑な文書理解 |
| チャットタイトル生成 | GPT-5.4 Nano | 単純要約 |
| タグ自動提案 | GPT-5.4 Nano | 単純分類 |

すべてVision対応モデル。Streaming を全面的に採用。

## 6. リリース計画

### 6.1 α版(招待制10人、無料)

- 捨てドメイン(.xyz等)で運用
- Vercel無料枠
- Google OAuth Testing Mode(テストユーザーとして手動追加)
- Stripe Test Mode(決済フローは動作するが実決済なし)
- プライバシーポリシー・利用規約はプレースホルダー
- Redeemコードで運用、全員に初期Credits付与
- 期間: 2〜4週間、フィードバックを収集

### 6.2 β版(限定公開、有料)

- 本ドメイン取得・移行(steadii.app等、空きを確認して決定)
- Google OAuth Production Mode への申請

## 8. リスクと緩和策

### 8.1 技術リスク
